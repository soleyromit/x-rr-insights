Faculty UX tension — power vs. casual why: informs progressive disclosure architecture decision
!
This is a cross-product signal. Same tension in FaaS form builder. Progressive disclosure is the platform-level architectural answer — not two separate interfaces.
Power user (30%)
Wants Bloom's tagging, LO linking, cross-course progression tracking, version comparison, usage analytics. Will invest time in configuration. Curriculum map is core value.
Research-first faculty (70%)
Teaching is secondary. Wants dead-simple question creation. No overhead. "Don't make me fill in more fields." Will abandon the platform if complexity threshold exceeded.
Design solution
Progressive disclosure: simple create flow by default. Power settings behind "advanced" toggle. AI handles tagging automatically so power users get value without burdening casual users.

Question Bank — Feature Tree
Full feature scope from Granola sessions · Versioned by delivery phase · AI layer annotated
Question creation
Rich text editor
Stem, options, image upload, media embed, table support. Rationale field (teacher-defined, shown post-submit on low-stakes).
Question types
MCQ · Multi-select · Clinical vignette (multi-paragraph) · True/False · Matching · Free response. V1 adds extended types.
Version control
Every edit creates new version. Owner approval required for modifications. Substantial changes = new question UUID. HOD sees version usage across faculty.
Tagging system
5–7 predefined categories
Question type · Bloom's (1–3) · Difficulty · Body system · Learning objective · NCCPA blueprint cell. Department-defined allowed values for strict categories.
AI tag assist ✦
Bloom's taxonomy auto-suggest on save. LO mapping suggestion. Tag consistency checker (heart vs cardiovascular). Import from ExamSoft auto-tags 10,000+ questions.
Smart Views
Personal private folders auto-populated by tag criteria. Mode 1: always-updated (latest versions). Mode 2: fixed snapshot (exam audit trail). Dynamic updates when new questions match.
Review workflow
Peer review
Async comment threading (V1 — not available today). Diff view showing changes between versions. Draft → In Review → Approved status states.
HOD approval
Head of department visibility into all version usage across faculty. Approval gates for new questions before campus-wide availability.
Import / migration
ExamSoft export → AI auto-tag → review → approve → bank. 10,000+ question migration supported. Folder structure auto-converted to flat tags.
Assessment assembly
Manual selection
Filter by any tag combination. Difficulty distribution dashboard. Preview questions before adding. Assessment creation from bank direct, within course, or during exam setup.
AI blueprint assembly ✦
V2: AI selects questions satisfying all NCCPA blueprint cells. 5-question Bloom's coverage check. Psychometric-based exam creation. "Lifesaver" — Touro direct quote.
Randomization
Question order randomization per student. Option order randomization. Parallel forms from same pool. Prevents cheating in synchronous settings.
Multi-campus sharing
Universal repository
Single source of truth — no more copy/paste between ExamSoft instances. View-only access for approved questions from other campuses. Build own exam from shared bank.
Access controls
Department / course-level access. View vs. modify rights. Campus-scoped visibility. Course-level auto-tagging when questions created within specific courses.
Approval workflow
New questions require campus approval before becoming available cross-campus. HOD can see which campuses are using which versions. Prevents unauthorized modifications.
Analytics layer
Item performance
p-value per question per course offering. Point biserial (closer to 1 = better). Discrimination index. Negative PBis = flag for review. Usage count across exams.
Clickable insights
Performance breakdown by course instance. Filter by creator, date range, difficulty. Correctness % across different student cohorts. Visual difficulty distribution dashboard.
Rubric support
% score rubrics for PD lab testing (Manhattan PA model). Faculty review questions as good/bad based on metrics. Integration with skills checklist for competency-mapped exams.

V1 AI — May 2026 sprint
✦ Bloom's tag assist on save
Automatic Bloom's level suggestion when faculty finishes writing a question. Faculty can accept, reject, or override. Rejection data used to refine model. Current accuracy: 50–60% → target 80%+.
✦ AI import tagger
ExamSoft 10,000+ question migration: AI auto-converts folder structure to flat tags. Tag consistency checker prevents drift (heart vs cardiovascular). Review queue for human approval before finalizing.
✦ Rationale generator
AI-assisted content for rationale field per question. Faculty review and edit. Shown post-submit on low-stakes exams. Confirmed V1 feature from Nipun session.
V2 AI — Jul–Aug 2026
✦ Blueprint-based exam assembly
AI selects questions from bank that collectively satisfy all NCCPA blueprint cells. 5-question Bloom's distribution check. Psychometric weighting. Direct quote from Touro: "Lifesaver." Must be live before Cohere Aug 2026.
✦ Question gen from slides
Faculty uploads lecture slides / course notes → AI generates draft questions. Faculty review, edit, approve. Bloom's levels applied automatically. Confirmed AI use case from Touro ExamSoft meeting.
✦ LO curriculum mapping
Cross-course knowledge dependency mapping. Identifies prerequisite gaps: students failing because an earlier course LO was not assessed. Current accuracy 50–60%, model refinement using acceptance data in progress.
V2+ AI — Nov–Dec 2026
✦ PANCE readiness predictor
"If student took PANCE today, probability of passing." Based on PACRAT, EOR, EOC scores + NCCPA blueprint coverage. Benchmark 75+ = good passing chance. PA program-specific. Confirmed from Aarti/Vishaka session.
✦ AI remediation personalizer
Based on where student fails (EOR topic, PACRAT domain) → generate 30-question PDF in PAEA format for that topic. Every student's remediation path is different. Direct Touro request. Saves faculty hours of manual assembly.
✦ Item anomaly alerts
Automated flag when p-value or point biserial falls outside acceptable range. Alert to faculty: "This question may need review." Negative PBis = low-knowledge students getting it right = question validity issue.

Decision 1: Flat tagging over folder hierarchy
Decision
Moved away from ExamSoft's rigid folder structure (master bank → live courses copy). Adopted flat tagging with Smart Views.
Rationale
ExamSoft folder structure fails cross-course knowledge mapping. Students fail because prerequisite knowledge gaps from other courses aren't tracked. Folders also break multi-campus sharing. Flat tagging with ~20 tags per question enables cross-course, cross-campus, and cross-cohort analytics.
Tradeoff considered
Folders feel familiar to faculty. Risk: cognitive overhead of tagging deters casual users. Mitigation: AI handles tagging on save. Smart Views recreate the folder UX on top of flat architecture.
Platform strategy · Mar 4 · Q-Bank architecture · Mar 12
Decision 2: Platform-embedded accessibility (no 3rd-party)
Decision
All accessibility features must be built into the exam platform. No external tools, screen readers, or OS-level affordances.
Rationale
Lockdown browser blocks all external tools. This is not a design preference — it is the only compliant path. Pearson Education model (GRE, SAT, TOEFL) proves it is achievable: their full accessibility suite is platform-embedded.
Tradeoff considered
Building custom OSK, zoom, TTS is significantly more expensive than relying on OS tools. Phased delivery (V0 → V1) manages this cost. V0 ships minimum viable accessibility for UNF pilot compliance. Full suite in V1.

Decision 3: React front-end, AI-first architecture
Decision
React front-end confirmed for entire app (admin + student). AI-first design philosophy: every new screen must consider AI integration upfront.
Rationale
Retrofitting AI into existing functionality is architectural debt. The PRISM Day 1 insight: AI must be designed in from scratch, not inserted. This is Exxat's primary competitive moat vs. ExamSoft (which cannot AI-first its 20-year-old codebase).
Tradeoff considered
AI features blocked in Phase 1 due to faculty adoption concerns. Risk of over-promising AI timeline. Mitigation: Phase 1 is AI-architecture-ready but AI-features-off. Phase 2 (May sprint) turns them on.
PRISM Day 2 · Mar 3 · Exam stand-up · Mar 19
Decision 4: Assessment creation as primary workflow
Decision
Current wireframe described as "too database-heavy." Assessment creation must be the primary workflow, not question bank management.
Rationale
Faculty's primary job is building and delivering exams, not curating a database. The question bank is infrastructure that supports assessment creation, not the other way around. 3 entry points to questions: bank direct, within course, during exam assembly. Assessment creation surfaces questions — not the other way.
Tradeoff considered
Power users (program-level) want to manage the bank as a primary workflow (HOD, curriculum director). Progressive disclosure: assessment creation as default entry; bank management in "advanced" or dedicated bank view for power users.
